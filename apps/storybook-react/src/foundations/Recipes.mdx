# Recipes

Complete, copy-paste page templates. Each recipe is a working file — drop it into a fresh
React or Svelte app, import `@w5-ui/<framework>/styles.css`, set `<html data-motif="dark">`,
and it renders pixel-for-pixel like the matching Dashboard story.

If you are wiring a brand new userland app, copy from here first; tune later. The recipes
deliberately favour the canonical components (`AppLayout`, `PageHeader`, `Card`, `Stat`,
`Table`, `field-stack`) over hand-rolled markup — that is the difference between userland
that matches and userland that drifts.

> Every recipe assumes `@w5-ui/react/styles.css` (or the Svelte / WC equivalent) is imported
> once at the app entry point and `<html data-motif="dark">` is set inline in `index.html`.
> See **Foundations / Setup** for the entry-file walkthrough.

## Recipe 1 — Dashboard overview

A landing page: hero stats across the top, two wide charts underneath, one full-width tile
at the bottom. Mirrors the **Dashboard** story tile-for-tile.

```tsx
// src/pages/Dashboard.tsx — React
import {
  AppLayout, Topbar, Sidebar, PageHeader,
  Stat, Card, CardTitle, Grid,
  Sparkline, LineChart, BarChart,
} from '@w5-ui/react'

export function Dashboard() {
  return (
    <AppLayout
      pageLabel="Dashboard"
      topbar={<Topbar siteName="HQ" activeApp="system" />}
      sidebar={<Sidebar sections={SECTIONS} activeId="dashboard" />}
    >
      <PageHeader title="Dashboard" />
      <Grid>
        <Stat span={4} label="Internet" value="847" unit="Mbps · down"
              sub="18 ms · 0.0% loss" delta="+12% 24h" deltaDir="up" />
        <Stat span={4} label="Clients" value="312" unit="active"
              sub="42 wired · 270 wireless" delta="+4 24h" deltaDir="up" />
        <Stat span={4} label="Alerts" value="3" unit="open"
              sub="1 critical · 2 warn" delta="-2 24h" deltaDir="down" />

        <Card span={8}>
          <CardTitle>Throughput</CardTitle>
          <LineChart data={throughput} />
        </Card>
        <Card span={4}>
          <CardTitle>Top clients</CardTitle>
          <Sparkline data={top} />
        </Card>

        <Card span={12}>
          <CardTitle>Traffic by device</CardTitle>
          <BarChart data={traffic} />
        </Card>
      </Grid>
    </AppLayout>
  )
}
```


**What carries the look**: `AppLayout` owns the shell padding. `PageHeader` owns the sticky
header rhythm. `<Grid>` owns the 12-column gutter (typed equivalent of `<div className="grid">`).
`Stat` and `Card span={n}` own tile padding; `<CardTitle>` is the typed sibling of `<h3>` so
the `.card h3` rhythm survives codegen, MDX, or stray wrapper `<div>`s. Nothing else needs to be set.

## Recipe 2 — List page with table

Devices-style page: page header with primary action + search, table with sortable headers,
pagination underneath, bottom selection toolbar when rows are checked. The selection
toolbar triggers `.grid:has(.sel-toolbar)` automatically — no extra padding required.

```tsx
// src/pages/Devices.tsx — React
import {
  AppLayout, Topbar, Sidebar, PageHeader,
  Card, Grid, Button, SearchBox, Table, SortHeader,
  TableShell, Pagination, SelectionToolbar, Checkbox,
} from '@w5-ui/react'

export function Devices() {
  const [selected, setSelected] = useState<string[]>([])
  return (
    <AppLayout pageLabel="Devices" topbar={<Topbar … />} sidebar={<Sidebar … />}>
      <PageHeader
        title="Devices"
        actions={
          <>
            <SearchBox placeholder="Search devices…" />
            <Button variant="primary">Adopt</Button>
          </>
        }
      />
      <Grid>
        <Card span={12}>
          <TableShell>
            <Table>
              <thead>
                <tr>
                  <th scope="col"><Checkbox … /></th>
                  <SortHeader sort="asc">Name</SortHeader>
                  <th scope="col">Model</th>
                  <th scope="col" className="mac">MAC</th>
                  <SortHeader>Clients</SortHeader>
                </tr>
              </thead>
              <tbody>{rows.map(r => <DeviceRow key={r.id} row={r} />)}</tbody>
            </Table>
          </TableShell>
          <Pagination page={page} total={total} onChange={setPage} />
        </Card>

        {selected.length > 0 && (
          <SelectionToolbar count={selected.length} onClear={() => setSelected([])}>
            <Button>Move</Button>
            <Button variant="danger">Forget</Button>
          </SelectionToolbar>
        )}
      </Grid>
    </AppLayout>
  )
}
```

**What carries the look**: `<Grid>` is the typed page grid. `<TableShell>` wraps `<Table>`
for the inset rhythm (typed sibling of `<div className="table-shell">`). `<th scope="col">`
carries a11y. `SortHeader` owns the sort icon + `aria-sort`. The selection toolbar is
rendered _inside_ `<Grid>` so the `.grid:has(.sel-toolbar)` rule reserves room for it —
never as a sibling of `<main>` or a fixed-positioned overlay you styled yourself.

## Recipe 3 — Settings / form page

Two-column form with grouped sections. `<FieldStack>` owns the 16 px vertical rhythm,
`<FormRow>` owns the 14 px column gap, `<FormActions>` owns the right-aligned button row.
Drop in `<Field label hint error required>` for every input.

```tsx
// src/pages/Settings.tsx — React
import {
  AppLayout, Topbar, Sidebar, PageHeader,
  Card, CardTitle, Field, FieldStack, FormRow, FormActions,
  Input, Select, Toggle, Button, Callout,
} from '@w5-ui/react'

export function Settings() {
  return (
    <AppLayout pageLabel="Settings" topbar={<Topbar … />} sidebar={<Sidebar … />}>
      <PageHeader
        title="Settings"
        actions={<Button variant="primary" type="submit" form="settings">Save</Button>}
      />
      <form id="settings" className="grid" onSubmit={onSave}>
        <Card span={6}>
          <CardTitle>Site</CardTitle>
          <FieldStack>
            <Field label="Site name" required hint="Visible in the topbar">
              <Input value={name} onChange={…} />
            </Field>
            <FormRow>
              <Field label="Country"><Select … /></Field>
              <Field label="Timezone"><Select … /></Field>
            </FormRow>
            <Field label="Maintenance window" hint="Daily 04:00 local">
              <Toggle checked={maint} onChange={…} />
            </Field>
          </FieldStack>
        </Card>

        <Card span={6}>
          <CardTitle>Network</CardTitle>
          <Callout variant="info" title="Restart required">
            Changes to LAN subnet require a controller reboot.
          </Callout>
          <FieldStack>
            <Field label="LAN subnet" error={cidrErr}>
              <Input value={cidr} onChange={…} />
            </Field>
            <Field label="MTU"><Input value={mtu} onChange={…} /></Field>
          </FieldStack>
          <FormActions>
            <Button>Cancel</Button>
            <Button variant="primary" type="submit">Apply</Button>
          </FormActions>
        </Card>
      </form>
    </AppLayout>
  )
}
```

**What carries the look**: the form itself _is_ the grid — no extra wrapper. `<CardTitle>`
keeps the section-title rhythm under the `.card h3` rule without leaking a raw `<h3>` into
copy-paste templates; `<FieldStack>` ladders fields with 16 px gaps; `<FormRow>` two-columns
inputs without bespoke gap overrides; `<FormActions>` right-aligns and adds the 16 px top
offset. Replace these typed primitives with hand-rolled flex containers and the rhythm drifts
within one PR. The underlying CSS hooks (`.card h3`, `.field-stack`, `.form-row`,
`.form-actions`) ship for parity, but the typed primitives are the preferred path — they are
auto-importable and survive a class-name rename without touching userland.

## Empty-state / loading shell

When data is still loading or empty, render the same shell so layout doesn't jump:

```tsx
<AppLayout pageLabel="Devices" topbar={…} sidebar={…}>
  <PageHeader title="Devices" />
  <Grid>
    <Card span={12}>
      {loading
        ? <Skeleton lines={6} />
        : items.length === 0
          ? <EmptyState title="No devices yet" hint="Adopt one to get started" />
          : <Table>…</Table>}
    </Card>
  </Grid>
</AppLayout>
```

`<Skeleton>` and `<EmptyState>` both reserve the same vertical room as the eventual table,
so the page never jolts when data arrives. Never conditionally render the `<Card>` itself —
the grid lane goes with it and your other tiles reflow.

## What every recipe omits (on purpose)

- **No inline `style={{ padding, gap }}`.** Everything spatial comes from `.grid` / `.card`
  / `.field-stack` / `.form-row` / `.form-actions`.
- **No raw hex colours.** Status comes from `Pill` / `Callout` / `Banner`; copy comes from
  `var(--text-1)` and `var(--text-2)`.
- **No bespoke page header.** `PageHeader` (or the `.ph-bar` / `.ph-title` / `.ph-actions`
  primitives) owns the sticky border + padding.
- **No wrapper `<div>` around `.grid`.** The grid is a direct child of `<main>` (which
  `AppLayout` renders for you). Wrappers nullify the 16 × 24 page gutter.

If a recipe needs to bend one of those rules, that delta belongs in **Foundations** as a
new primitive — not as a one-off override in userland. Surface it instead of hand-rolling.
