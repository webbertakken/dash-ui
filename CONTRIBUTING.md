# Contributing to Dash UI

Thanks for your interest in contributing!

## Local setup

Prerequisites: Node 22+, yarn 4+.

```sh
git clone https://github.com/webbertakken/dash-ui.git
cd design-system
yarn install
yarn test          # ~1,300 tests across React, Svelte and Web Components
yarn build         # build all packages
yarn build:site    # build the static tri-framework Storybook site
yarn serve:site    # serve dist/ at http://localhost:4173
```

## Branching and PRs

- Branch off `main`
- Open a PR; CI will run tests, typecheck, and build all three Storybooks
- `main` is protected: squash-merge only, no force push, no deletion
- Small focused PRs preferred over big batches

## Commit messages

Imperative mood, present tense, under 72 chars on the first line:

```
Add LollipopChart variant with vertical layout
Fix focus-visible ring on Toggle in light motif
```

If the change is bigger than one sentence, add a body with the why.

## Adding a component

A new component lives in **three** places:

1. `packages/react/src/components/<Name>.tsx`
2. `packages/svelte/src/lib/components/<Name>.svelte`
3. Fixture entry in `packages/react/test-fixtures/fixtures.tsx` and `packages/svelte/test-fixtures/props.ts`

The Web Components layer is auto-generated by `yarn workspace @dash-ui/wc generate` from the Svelte source.

Update `packages/storybook-meta/src/categories.ts` so the new component is grouped under one of the 12 sidebar categories.

## Tests

- Unit + render smoke tests: vitest. New components must have entries in both fixtures registries.
- SSR smoke: `react-dom/server.renderToString` (React) and `generate: 'ssr'` compile (Svelte) run on every variant - failing SSR will fail CI.

## Code style

- TypeScript everywhere
- Prefer pure functions
- Explicit prop types; no `any` without a comment explaining why
- One file per component; sub-components stay in the same file unless they ship on their own (e.g. `Accordion` + `AccordionItem`)

## Releasing

Maintainers only. Uses changesets (forthcoming) to publish `@dash-ui/*` packages independently.

## Questions

Open a discussion or issue. Substantive design changes should be proposed as an issue with a multi-choice for the alternatives.
