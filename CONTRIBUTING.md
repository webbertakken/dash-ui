# Contributing to Dash UI

Thanks for your interest in contributing!

## Local setup

Prerequisites: Node 22+, yarn 4+.

```sh
git clone https://github.com/webbertakken/dash-ui.git
cd design-system
yarn install
yarn lint          # oxlint (~1s) - report-only on warnings
yarn format:check  # oxfmt - byte-stable formatter
yarn typecheck     # tsgo --noEmit on the shipping packages
yarn test          # ~1,300 tests across React, Svelte and Web Components
yarn build         # build all packages
yarn build:site    # build the static tri-framework Storybook site
yarn serve:site    # serve dist/ at http://localhost:4173
```

## Branching and PRs

- Branch off `main`
- Open a PR; CI will run tests, typecheck, and build all three Storybooks
- `main` is protected: squash-merge only, no force push, no deletion
- Small focused PRs preferred over big batches

## Commit messages

Imperative mood, present tense, under 72 chars on the first line:

```
Add LollipopChart variant with vertical layout
Fix focus-visible ring on Toggle in light motif
```

If the change is bigger than one sentence, add a body with the why.

## Adding a component

A new component lives in **three** places:

1. `packages/react/src/components/<Name>.tsx`
2. `packages/svelte/src/lib/components/<Name>.svelte`
3. Fixture entry in `packages/react/test-fixtures/fixtures.tsx` and
   `packages/svelte/test-fixtures/props.ts`

The Web Components layer is auto-generated by `yarn workspace @w5-ui/wc generate` from the Svelte
source.

Update `packages/storybook-meta/src/categories.ts` so the new component is grouped under one of the
12 sidebar categories.

## Tests

- Unit + render smoke tests: vitest. New components must have entries in both fixtures registries.
- SSR smoke: `react-dom/server.renderToString` (React) and `generate: 'ssr'` compile (Svelte) run on
  every variant - failing SSR will fail CI.

## Code style

- TypeScript everywhere
- Prefer pure functions
- Explicit prop types; no `any` without a comment explaining why
- One file per component; sub-components stay in the same file unless they ship on their own (e.g.
  `Accordion` + `AccordionItem`)

## Releasing

Maintainers only. We use [Changesets](https://github.com/changesets/changesets) to coordinate
versions across the `@w5-ui/*` packages.

```sh
yarn changeset            # interactive: pick packages, bump type, write a one-liner
git add .changeset && git commit -m "chore: changeset for <feature>"
```

What happens after the commit lands on `main`:

1. `.github/workflows/release.yml` opens / updates a **"chore: version packages"** PR. Merging that
   PR runs `yarn release`, publishes the bumped `@w5-ui/*` packages to npm with provenance, and
   pushes per-package tags shaped `@w5-ui/<pkg>@X.Y.Z`.
2. `.github/workflows/release-github.yml` watches for tags matching `**@*.0.0` and **only** creates
   a GitHub Release when the version is `X.0.0` with `X >= 1`. Minor and patch tags fire silently.
   Pre-1.0 (`0.x.y`) tags never produce a Release.

Followers therefore see one Release notification per major release across the whole `@w5-ui` suite,
not one per patch.

### Required repo secrets

- `NPM_TOKEN` - granular publish token scoped to `@w5-ui/*` with 2FA "Authorization only" mode.
  Generate at https://www.npmjs.com/settings/<user>/tokens.

### Linked packages

All five public packages (`react`, `svelte`, `wc`, `tokens`, `assets`) are `linked` in
`.changeset/config.json`. A `minor` bump on any of them bumps all five to the same minor version.
Keeps the cross-package import story sane.

## Questions

Open a discussion or issue. Substantive design changes should be proposed as an issue with a
multi-choice for the alternatives.
